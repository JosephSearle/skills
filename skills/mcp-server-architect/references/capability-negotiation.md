# Capability Negotiation Reference

## Protocol Version

Current MCP spec version: **2025-11-25**

Version strings use `YYYY-MM-DD` format. A new version is issued only on backwards-incompatible changes.

History: 2024-11-05 → 2025-03-26 (tool annotations, Streamable HTTP) → 2025-06-18 (auth separation) → 2025-11-25 (current).

---

## initialize Handshake

The client sends `initialize` as the first request:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-11-25",
    "capabilities": {
      "roots": { "listChanged": true },
      "sampling": {},
      "elicitation": {}
    },
    "clientInfo": { "name": "claude-desktop", "version": "1.2.3" }
  }
}
```

Server responds:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-11-25",
    "capabilities": {
      "tools": { "listChanged": false },
      "resources": { "listChanged": false, "subscribe": false },
      "logging": {}
    },
    "serverInfo": { "name": "my-mcp-server", "version": "1.0.0" }
  }
}
```

If no mutually compatible version exists, the server MUST terminate the connection with an error.

## Server Capabilities Reference

| Capability | Field | Meaning |
|-----------|-------|---------|
| `tools` | `listChanged` | Server will emit `notifications/tools/list_changed` when tool list changes |
| `resources` | `listChanged` | Server will emit `notifications/resources/list_changed` |
| `resources` | `subscribe` | Server supports `resources/subscribe` (stateful only) |
| `prompts` | `listChanged` | Server will emit `notifications/prompts/list_changed` |
| `logging` | — | Server accepts `logging/setLevel` and emits `notifications/message` |
| `completions` | — | Server supports `completion/complete` for argument autocompletion |
| `experimental` | key-value map | Non-standard extensions |

---

## Client Capabilities Reference

| Capability | Meaning | Impact on server |
|-----------|---------|-----------------|
| `sampling` | Client can handle `sampling/createMessage` requests | Server may request LLM completions from the host |
| `elicitation` | Client can handle `elicitation/create` requests | Server may request user input from the host |
| `roots` | Client provides root URIs via `roots/list` | Server can use roots for resource resolution |

---

## Declaration Rules

- ONLY declare capabilities you implement.
- MUST NOT emit `notifications/*/list_changed` unless `listChanged: true` is declared.
- Check client capabilities before invoking `sampling/createMessage` or `elicitation/create`.

## @rekog/mcp-nest Capability Config

```ts
McpModule.forRoot({
  name: 'my-server',
  version: '1.0.0',
  capabilities: {
    tools:     { listChanged: false },
    resources: { listChanged: false, subscribe: false },
    prompts:   { listChanged: false },
    logging:   {},
  },
})
```

If `capabilities` is omitted, `@rekog/mcp-nest` derives it from the registered providers automatically. Explicit declaration is recommended for auditability.

## JSON-RPC Error Codes

Standard reserved range -32768 to -32000:

| Code | Name | When to use |
|------|------|------------|
| -32700 | Parse error | Malformed JSON |
| -32600 | Invalid request | Not a valid JSON-RPC request |
| -32601 | Method not found | Unknown method name |
| -32602 | Invalid params | Zod schema rejection (auto-generated by SDK) |
| -32603 | Internal error | Unhandled server exception |
| -32002 | Resource not found | MCP addition |
| -32800 | Request cancelled | MCP addition — `notifications/cancelled` received |
| -32801 | Content too large | MCP addition |

Custom application codes: use range outside -32768 to -32000 (e.g., -31xxx for auth, -30xxx for domain).
