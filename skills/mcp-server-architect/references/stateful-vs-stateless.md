# Stateful vs Stateless Reference

## Quick Comparison

| Dimension | Stateless | Stateful |
|-----------|-----------|---------|
| `statelessMode` | `true` | `false` |
| `sessionIdGenerator` | not set | `() => randomUUID()` |
| `enableJsonResponse` | `true` (recommended) | `false` (SSE for notifications) |
| Load balancing | Any LB — no sticky sessions | Sticky sessions OR shared state store required |
| Horizontal scale | Drop-in | Needs session affinity or externalised state |
| Server-initiated messages | Not supported | Supported via `GET /mcp` SSE |
| `resources/subscribe` | Not supported | Supported |
| Sampling / elicitation | Not supported | Supported |
| Progress notifications mid-call | Not supported | Supported |

---

## Stateless Mode

Best default for tool-only servers.

```ts
streamableHttp: {
  statelessMode: true,
  enableJsonResponse: true,  // returns application/json — simpler for WAFs
}
```

Each `POST /mcp` is fully independent. No `Mcp-Session-Id` header. No `GET /mcp` SSE endpoint needed.

**JSON response mode** returns a single `application/json` body for non-streaming requests. Advantages:
- WAF and API Gateway compatible without special stream handling.
- Simpler client implementations.
- Easier load balancer health checks.

Limitation: cannot send mid-call `notifications/progress` on the same response. Progress on slow tools requires a stateful upgrade or a polling pattern.

---

## Stateful Mode

Required for sampling, elicitation, subscriptions, and resumable SSE.

```ts
import { randomUUID } from 'node:crypto';

streamableHttp: {
  statelessMode: false,
  sessionIdGenerator: () => randomUUID(),  // MUST be crypto-random
  enableJsonResponse: false,
}
```

**Session ID security rules:**
1. MUST be generated by `node:crypto` `randomUUID()` or equivalent CSPRNG — not `Math.random()`.
2. SHOULD include the user's identity to prevent cross-session injection across replicas:
   ```ts
   sessionIdGenerator: (req) => `${req.user?.sub ?? 'anon'}:${randomUUID()}`
   ```
3. Treat session IDs as secrets — do not log them, do not include in error messages.

**Replica deployment:** without sticky sessions, a second replica will reject `Mcp-Session-Id` it did not issue. Options:
- Nginx `ip_hash` or AWS ALB sticky sessions.
- Externalise session state to Redis and share across replicas.

---

## Capability Constraints by Mode

| Capability | Stateless | Stateful |
|-----------|-----------|---------|
| `tools` | Yes | Yes |
| `resources` (read) | Yes | Yes |
| `resources/subscribe` | **No** | Yes |
| `prompts` | Yes | Yes |
| `logging` (server → client) | **No** | Yes |
| `sampling` (server → client) | **No** | Yes |
| `elicitation` | **No** | Yes |
| Progress notifications | **No** | Yes |

---

## Choosing the Right Mode

```
Does the server need any of: sampling, elicitation, resources/subscribe, progress notifications?
  └─ YES → Stateful

Is the server deployed across multiple replicas without sticky sessions?
  └─ YES + stateless → go stateless; nothing to do
  └─ YES + stateful required → add sticky sessions or externalise session state

Is simplicity and horizontal scale the primary concern?
  └─ YES → Stateless
```
