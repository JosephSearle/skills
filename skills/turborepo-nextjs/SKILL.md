# turborepo-nextjs

Next.js is the most common application type in Turborepo monorepos. The integration is well-supported, but has one required step that almost every newcomer misses: **every internal JIT package consumed by a Next.js app must be listed in `transpilePackages`**. Missing this causes cryptic syntax errors at build time, not import errors.

---

## Core Philosophy

**`transpilePackages` is mandatory for JIT packages.** Next.js's bundler (webpack or Turbopack) needs to transpile TypeScript source from internal packages. Without `transpilePackages`, the bundler encounters `.ts`/`.tsx` syntax it wasn't told to handle and throws a syntax error. This is the most common Next.js monorepo misconfiguration.

**Framework Inference handles `NEXT_PUBLIC_*` automatically.** Turborepo detects Next.js and auto-hashes all `NEXT_PUBLIC_*` env vars — you don't need to manually list them in `turbo.json`'s `env` array. Server-side env vars still need explicit declaration.

## Step 1 — Detect Next.js setup

```
Locate apps/web/next.config.ts (or .js, .mjs):
  └─ Does it have "transpilePackages"?
       └─ NO and the app consumes internal JIT packages → add transpilePackages (Step 3)

Locate apps/web/turbo.json or root turbo.json "build" task:
  └─ Does "outputs" include [".next/**", "!.next/cache/**"]?
       └─ NO → fix outputs (Step 3)

Check .gitignore:
  └─ Is next-env.d.ts excluded?
       └─ NO → add it
```

## Step 3 — Execute

### Canonical Next.js app setup in turbo.json

```jsonc
// turbo.json
{
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "!.next/cache/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "start": {
      "dependsOn": ["build"]
    },
    "lint": {
      "outputs": []
    },
    "check-types": {
      "dependsOn": ["^check-types"]
    }
  }
}
```

`.next/cache/**` is excluded from outputs — it is Next.js's own incremental cache and should not be stored in the Turborepo cache (too large, separate lifecycle).

### Add `transpilePackages`

```ts
// apps/web/next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  transpilePackages: ["@repo/ui", "@repo/utils"],
};

export default nextConfig;
```

Every internal JIT package (`exports` points to `.ts`/`.tsx` source) must be listed. Compiled packages (exports point to `dist/`) do not need `transpilePackages`.

### Sharing `@repo/ui` — per-path exports pattern

```json
// packages/ui/package.json
{
  "name": "@repo/ui",
  "exports": {
    "./button": "./src/button.tsx",
    "./card": "./src/card.tsx",
    "./input": "./src/input.tsx"
  }
}
```

```tsx
// apps/web/src/app/page.tsx
import { Button } from "@repo/ui/button";
import { Card } from "@repo/ui/card";
```

Per-path exports enable tree-shaking and give each component a clean import path. Avoid barrel exports (`index.ts`) for large UI packages — they defeat tree-shaking.

### `next-env.d.ts`

This file is auto-generated by Next.js on every `next build` or `next dev` run. It must not be committed.

```gitignore
# .gitignore
next-env.d.ts
```

```jsonc
// apps/web/tsconfig.json — include it but don't commit it
{
  "extends": "@repo/typescript-config/nextjs.json",
  "include": ["next-env.d.ts", "**/*.ts", "**/*.tsx"],
  "exclude": ["node_modules"]
}
```

## Step 4 — Validate

- [ ] Every internal JIT package listed in `transpilePackages`
- [ ] Compiled packages (with `dist/`) are NOT in `transpilePackages` (not needed)
- [ ] `turbo.json` `build` outputs include `.next/**` and exclude `!.next/cache/**`
- [ ] `next-env.d.ts` is in `.gitignore`
- [ ] `next-env.d.ts` is included in `tsconfig.json`
- [ ] Server-only env vars (non-`NEXT_PUBLIC_*`) are in `turbo.json` `env` array
- [ ] For standalone output: `outputFileTracingRoot` points to the monorepo root, not `apps/web`
