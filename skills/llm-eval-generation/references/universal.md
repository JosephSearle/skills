# Universal LLM Eval Generation Reference

Apply this reference to all generated evals, regardless of scenario or framework.

---

## Governing Standards

### ISO/IEC 42001:2023 — AI Management System
The first global standard for AI governance (analogous to ISO 27001 for security). Does not
prescribe evaluation techniques but requires organisations to establish structured processes
around AI risk — eval pipelines must fit within this governance structure. Certifiable via
external audit. Annex A provides the reference control set for AI-specific risks.

### NIST AI RMF 1.0 + AI 600-1 GenAI Profile
Voluntary but widely adopted; referenced in US federal policy. Four core functions:

| Function | Eval implication |
|---|---|
| GOVERN | Policies for eval cadence, thresholds, ownership |
| MAP | Categorise risks per LLM use case and context |
| MEASURE | Quantitative and qualitative assessment of AI risks |
| MANAGE | Prioritise, respond to, and monitor identified risks |

**NIST AI 600-1** (July 2024) is the GenAI-specific profile. It identifies these LLM risks as
requiring direct measurement: hallucination and confabulation, prompt injection, training data
poisoning, sensitive information leakage, and homogenisation (over-reliance on a single model).

### EU AI Act (In Force August 2024; GPAI Obligations August 2025)
Categorises AI systems by risk tier. For GPAI models (>10²⁵ FLOPs training): adversarial
testing and incident reporting are required. For limited-risk systems (chatbots): transparency
obligations apply. Eval suites must be documented and version-controlled to support conformity
assessment under high-risk classification.

### OECD AI Principles (Updated 2024)
Five principles adopted by 40+ countries. Three directly imply eval requirements:
- **Transparency and explainability** → output-level evaluation
- **Robustness, security, and safety** → adversarial testing
- **Accountability** → audit trails and eval documentation

## HELM — 7 Evaluation Dimensions

Stanford's HELM (Holistic Evaluation of Language Models) established the canonical taxonomy for
what LLM evaluation should measure. Every scenario-specific eval should address the dimensions
that apply to it.

| Dimension | Definition | Primary evaluation method |
|---|---|---|
| Accuracy | Correctness of outputs against ground truth | Task-specific metrics (EM, F1, pass@k) |
| Calibration | Model confidence matches actual accuracy | Expected Calibration Error (ECE) |
| Robustness | Consistency under input perturbations | Accuracy on paraphrased vs. original inputs |
| Fairness | Equal performance across demographic groups | Performance gap across subgroups |
| Bias | Systematic skew toward stereotypes or groups | WinoBias, BBQ benchmark |
| Toxicity | Harmful, offensive, or dangerous outputs | Perspective API, ToxiGen |
| Efficiency | Compute cost per unit of performance | Tokens/second, cost-per-query, latency |

## Core Metric Taxonomy

### Factuality & Groundedness

**Faithfulness** — whether claims in the output are supported by the provided context (source
documents). Calculated as: supported claims ÷ total claims. Critical for RAG. Does NOT require
ground truth labels — evaluated against retrieved context only.

**Groundedness** — stricter than faithfulness. Every statement must have traceable attribution
in the source. Requires per-claim verification.

**Factuality** — accuracy with respect to real-world facts, not just provided context. Requires
external knowledge base or human verification.

> Faithfulness ≠ Factuality. A response can be faithful to a wrong document.

**Hallucination** — information generated by the model not present in or contradicted by source
material. Two types:
- Intrinsic: contradicts the source context
- Extrinsic: introduces information not in the source (may or may not be factually correct)

### Relevance

**Answer Relevance** — whether the response addresses what the user actually asked. Measured via
semantic similarity between response and query, or LLM-as-judge scoring.

**Context Relevance** (RAG-specific) — whether retrieved documents relate to the query.
Irrelevant retrieval caps generation quality regardless of model capability.

**Context Precision** — what fraction of retrieved context is actually useful for the answer.

**Context Recall** — what fraction of the information needed to answer was retrieved.

### Quality

**Coherence** — logical consistency and non-contradictory structure within the response.

**Fluency** — grammatical and linguistic quality.

**Conciseness / Completeness** — answers fully without unnecessary padding, or conversely,
without omitting critical information.

**Helpfulness** — whether the output actually helps the user accomplish their goal. Typically
requires LLM-as-judge or human evaluation.

### Safety

**Toxicity** — presence of harmful, offensive, or hateful content.

**Bias** — systematic stereotyping or unequal treatment of groups.

**Privacy** — leakage of PII or sensitive information from training data or context.

**Prompt Injection Resistance** — robustness against adversarial instruction override.

## Evaluation Methodologies

### Reference-Based
Compares model output to a ground-truth reference answer. Requires labelled data.

| Metric | Use case | Notes |
|---|---|---|
| Exact Match (EM) | Classification, structured QA | Binary; strict |
| F1 | Open-domain QA | Token-level overlap |
| BLEU / ROUGE | Summarization, translation | Weak proxy; poor human alignment |
| BERTScore | Semantic similarity | Better human alignment than BLEU |

### Reference-Free
Does not require labelled ground truth. Scales to large datasets.

**LLM-as-Judge** — a separate LLM evaluates output quality using a rubric. Most scalable for
subjective criteria. Highest correlation with human judgment when configured correctly (see
LLM-as-Judge Standards below).

**G-Eval** — LLM-as-judge with explicit Chain-of-Thought reasoning steps before scoring.
Better calibration than direct scoring. From: "NLG Evaluation using GPT-4 with Better Human
Alignment" (Liu et al., 2023).

**SelfCheckGPT** — sampling-based; assumes hallucinated outputs are not reproducible across
multiple samples. Hallucinations are identified by inconsistency across samples.

### Execution-Based (Code)
**pass@k** — probability that at least one of k generated solutions passes all tests. Industry
standard for code generation (HumanEval, SWE-bench). Significantly more reliable than static
analysis.

### Human Evaluation
Gold standard for subjective quality. Used to validate automated evals. Expensive — reserve for
calibrating automated judges and reviewing edge cases.

## LLM-as-Judge Standards

### Design Requirements

**Explicit rubric** — define criteria precisely before writing the judge prompt. Vague criteria
produce noisy, inconsistent scores that cannot be reproduced or improved.

```
# Example rubric for answer relevance
Score 1: The response does not address the question at all
Score 3: The response partially addresses the question but omits key aspects
Score 5: The response fully and directly addresses the question
```

**Chain-of-Thought before scoring (G-Eval)** — require the judge to reason through the criteria
before outputting a score. Improves calibration and score consistency.

```
Evaluate the following response on the criterion of faithfulness.

Step 1: Identify each factual claim in the response.
Step 2: Check whether each claim is supported by the provided context.
Step 3: Calculate the ratio of supported claims to total claims.
Step 4: Assign a score from 1–5 based on this ratio.

Output your evaluation as JSON:
{"reasoning": "<your step-by-step analysis>", "score": <1-5>}
```

**Structured output** — always require `{"score": N, "rationale": "..."}`. Free-form text
output cannot be parsed reliably for CI gating.

**Judge model selection** — use a different, more capable model as judge than the model under
evaluation. A model judging its own outputs exhibits self-preference bias.

**Calibration validation** — before using a judge in CI, validate it against a human-labelled
sample. Target Pearson correlation > 0.8 with human ratings. A judge prompt that has not been
validated against human labels is an unverified assumption.

### Judge Failure Modes

| Failure mode | Description | Mitigation |
|---|---|---|
| Sycophancy | Favours verbose, confident-sounding responses over accurate ones | Include explicit anti-sycophancy instruction in rubric |
| Self-preference bias | A model rates its own outputs higher | Use a different model as judge |
| Length bias | Longer responses score higher regardless of quality | Penalise padding in rubric; include conciseness criterion |
| Positional bias | In A/B comparisons, favours the first option presented | Randomise order; run both orderings and average |
| Prompt sensitivity | Scores shift with minor rubric rewording | Test rubric stability before deploying to CI |

## Eval Dataset Design Standards

**Minimum size:** 10–20 cases per skill or scenario is sufficient to surface regressions early
(OpenAI Skill Eval Framework). Scale up when early effect sizes are small or the scenario is
high-risk.

**Required composition:**

| Category | Description |
|---|---|
| Happy path | Typical inputs with known-good expected outputs |
| Edge cases | Boundary inputs, unusual formats, minimal valid inputs |
| Adversarial | Inputs designed to exploit failure modes |
| Negative / refusal | Cases where the correct answer is to decline, abstain, or escalate |

**Source priority:**
1. Production failures converted directly to test cases — these are the most valuable
2. Real user queries (anonymised)
3. Synthetic generation, reviewed by a human before inclusion

Do not build eval sets from synthetic data alone. Synthetic data encodes the model's own biases.

**Versioning:** treat `evals/datasets/` like source code. Version-control eval datasets
alongside prompt and model changes. Include a changelog entry when cases are added or modified.
Every prompt change should trigger a full eval run against the versioned dataset.

**Benchmark contamination warning** (2025): models trained on post-cutoff data can score high
on static benchmarks without genuine capability. Use held-out private eval sets; track
production metrics alongside benchmark scores; prefer dynamic evals over static datasets for
CI gating.

**Ambiguity standard (OpenAI Skill Eval Framework):** design tasks that two domain experts
would grade identically. Ambiguous tasks produce noisy metrics. A 0% pass rate almost always
signals a broken task, not an incapable model.

## CI/CD Eval Pipeline Architecture

```
Development → Offline Eval → Staging → Online Eval → Production Monitoring
```

**Offline Eval (pre-deployment):**
- Run against a fixed, versioned golden dataset
- Reference-based metrics where ground truth exists
- LLM-as-judge for subjective quality dimensions
- Regression gate: block deploy if score drops below threshold vs. baseline
- Run with ≥ 3 trials per case to account for model stochasticity

**Online Eval (production sampling):**
- Sample 5–10% of live traffic for continuous evaluation
- Real user queries as eval inputs
- Monitor for distribution shift between offline eval set and live traffic
- Alert on metric drift beyond configured thresholds

**Regression tracking:**
- Version eval datasets alongside prompts and models
- Every prompt change triggers a full eval run
- Track scores over time — visualise trend, not just point-in-time value
- Store baseline results; compare new runs against baseline with statistical significance

**Default CI thresholds** (override if project configures its own):

| Metric | Minimum threshold |
|---|---|
| Faithfulness | 0.8 |
| Answer relevance | 0.7 |
| Context precision | 0.7 |
| Custom G-Eval (1–10 scale) | 7.0 |
| pass@1 (code generation) | 0.8 |
| Tool selection accuracy | 0.9 |
